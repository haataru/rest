# Архитектура Проекта Rest

Проект **Rest** представляет собой независимый экосистемный стэк для разработки высоконагруженных веб-приложений. 
Его главная цель — предоставить разработчикам скорость C/C++ (за счет компиляции через LLVM) и удобство скриптовых языков (благодаря автоматическому управлению памятью и веб-ориентированному синтаксису).

## Глобальное разделение (Cargo Workspace)

На данный момент проект физически разделен на два основных крейта (cargo workspace), которые в будущем будут объединены, когда язык станет полностью независимым:

1. **`compiler/` (Крейт `restc`)**
   Содержит исходный код самого компилятора языка Rest.
   - Написан на Rust.
   - Отвечает за полный цикл компиляции исходного `.rest` кода: от лексического анализа до генерации машинного кода через LLVM (`inkwell`).

2. **`runtime/` (Крейт `rest_runtime`)**
   Содержит среду выполнения и стандартную библиотеку для скомпилированных `.rest` программ.
   - Сейчас включает в себя системный аллокатор, механизмы ARC (Automatic Reference Counting) и веб-сервер на базе Tokio/Hyper.
   - **Глобальная цель:** Полностью переписать `runtime/` на самом языке Rest (начиная с сокетов и HTTP-роутинга) и отказаться от промежуточного слоя на Rust.

## Жизненный цикл компиляции

Когда вы запускаете компилятор (например, `cargo run -- web_magic.rest`), происходит следующий конвейер:

1. **Лексический анализ (Lexer):**
   Исходный текст разбивается на токены (ключевые слова, идентификаторы, операторы).

2. **Синтаксический анализ (Parser):**
   Токены собираются в Абстрактное Синтаксическое Дерево (AST). Здесь распознаются функции, структуры, веб-декораторы (`@get`, `@post`) и выражения.

3. **Семантический анализ (TypeChecker):**
   Проверка типов (строгая типизация), проверка корректности L-Value для указателей (`&x`), а также проверка возможности приведения типов (кастов).

4. **Lowering (Понижение в HIR):**
   AST трансформируется в High-Level Intermediate Representation (HIR). На этом этапе раскрываются "синтаксические сахара", например `array.length()` или неявные касты.

5. **Генерация LLVM IR (Codegen):**
   Компилятор переводит HIR в LLVM IR. 
   - Здесь автоматически вставляются вызовы функций управления памятью (`__rest_retain`, `__rest_release`).
   - Генерируются HTTP-обертки для функций, помеченных декораторами.

6. **Компиляция и Линковка:**
   LLVM оптимизирует IR (passes) и генерирует объектный файл (`output.o`). После этого вызывается системный `cc` (C compiler/linker), который линкует объектный файл с нашим рантаймом `librest_runtime.a` и сторонними библиотеками, производя готовый исполняемый бинарник.

## Взаимодействие с ОС (Системные вызовы)

Мы идем к тому, чтобы язык мог общаться с операционной системой напрямую (Zero-Dependency). Для этого компилятор имеет встроенную поддержку C-совместимого **FFI** (Foreign Function Interface):
- Ключевое слово `extern fn` позволяет декларировать внешние системные вызовы.
- Наличие сырых указателей (`*T`, `&`) позволяет работать с системной памятью и C-структурами (например, `sockaddr`).

## Интеграция с Web (Встроенный сервер)
Язык разрабатывался специально для веба. Разработчик пишет обычные функции, помечает их декораторами (`@get("/api")`), а компилятор **самостоятельно**:
1. Генерирует C-совместимые функции-обертки, принимающие HTTP-запросы.
2. Инициализирует глобальный роутинг.
3. Добавляет неявный вызов `rest_start_server` в начало программы (если есть маршруты).
