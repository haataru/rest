# Устройство Компилятора (Compiler Internals)

Компилятор Rest (`restc`) написан на Rust и разбит на изолированные модули, образуя четко выстроенный конвейер трансляции (Pipeline). 
Код компилятора находится в крейте `compiler/`.

## 1. Пайплайн Компиляции

Компиляция исходного файла проходит следующие стадии:
`Source Code (.rest)` → `Lexer` → `Parser` → `AST` → `Semantic Analyzer` → `Lowering` → `HIR` → `LLVM Codegen` → `Object File (.o)`

## 2. Модули и их функции

### `lexer/` (Лексический Анализатор)
Читает исходный текст посимвольно и генерирует поток `Token`. 
- Обрабатывает ключевые слова, операторы (`+`, `-`, `&`, `*`, `as`), числа, строковые литералы и идентификаторы.
- Игнорирует пробелы и комментарии.

### `parser/` (Синтаксический Анализатор)
Использует метод рекурсивного спуска (Recursive Descent Parser) для построения `AST` (Abstract Syntax Tree).
- **Выражения (Expressions):** Распознает приоритеты операторов (Pratt Parsing).
- **Утверждения (Statements):** Распознает структуры (`struct`), функции (`fn`), внешние функции (`extern fn`), циклы и присваивания.
- **Декораторы:** Парсит аннотации `@get`, `@post` и привязывает их к функциям.

### `sema/` (Semantic Analyzer / TypeChecker)
Выполняет строгую проверку типов перед тем, как переходить к генерации кода. 
- Проверяет существование переменных и функций.
- Гарантирует совпадение типов при присваиваниях и вызовах функций.
- **Проверки указателей:** 
  - Разрешает взятие адреса `&` только от *l-value* выражений (переменные, поля структур, элементы массивов, разыменованные указатели).
  - Проверяет совместимость типов при использовании оператора приведения `as` (разрешены конвертации: int <-> int, ptr <-> ptr, int <-> ptr).

### `ir/` (Intermediate Representation)
Здесь AST понижается (lowering) в `HIR` (High-Level Intermediate Representation).
- `lower.rs` — преобразует AST узлы в HIR узлы. На этом этапе обрабатываются неявные детали языка (например, дешугаризация `.length()` для массивов).
- Модуль описывает внутреннее представление функций, блоков, выражений (включая `AddressOf`, `Dereference`, `SizeOf`, `Cast`) и внешних функций.

### `codegen/` (Генерация LLVM IR)
Самый массивный и важный этап, где HIR транслируется в машинонезависимый LLVM IR.
Для чистоты кода этот этап разбит на несколько подмодулей:
- **`compile.rs`:** Основная обертка (Codegen struct). Инициализирует функции рантайма (`malloc`, `free`, `__rest_retain`), обрабатывает функции и глобальный роутинг веб-сервера.
- **`expr.rs`:** Компиляция выражений (математика, вызовы функций, массивы). Здесь реализован `compile_lvalue_ptr` — функция, которая умеет вычислять точный адрес в памяти для вложенных структур и массивов, чтобы реализовать оператор взятия адреса `&`.
- **`stmt.rs`:** Компиляция утверждений (if/else, while, переменные, return). Обрабатывает создание аллокаций на стеке (`alloca`).
- **`memory.rs` / `types.rs`:** Работа с типами LLVM (преобразование `HIR Type` в `BasicTypeEnum` LLVM) и управление памятью (вставка вызовов ARC для освобождения).

## 3. Обработка Внешних Функций (FFI)
Когда в коде встречается `extern fn foo(a: int) -> int;`:
1. `Parser` создает `AstStmt::ExternFn`.
2. `Sema` регистрирует её в таблице функций.
3. `Lowering` конвертирует её в `HirStmt::ExternFn`.
4. `Codegen` в цикле компиляции перехватывает её и вызывает `self.module.add_function(name, type, None)`. 
Отсутствие тела у функции говорит LLVM, что это внешняя декларация (`declare`), и линкер позже свяжет этот вызов с системной библиотекой (libc) или нашим рантаймом.

## 4. Особенности LLVM и Поинтеров (LLVM-15+ Opaque Pointers)
Начиная с версии LLVM 15, все указатели стали "непрозрачными" (`ptr`). Наш компилятор использует Inkwell 0.9, адаптированный под LLVM 18.
Поэтому `compile_lvalue_ptr` и аналогичные функции всегда возвращают `PointerValue`, а для обращения к памяти (`load`/`store`) или навигации (`GEP` - GetElementPtr) мы обязаны явно передавать тип (Type), с которым работаем в данной ячейке памяти.
