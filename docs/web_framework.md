# Веб-Фреймворк и Роутинг

В языке Rest веб-фреймворк — это не сторонняя библиотека, а встроенный элемент языка (First-Class Citizen). Это позволяет достичь феноменальной производительности (по нашим бенчмаркам около 290k RPS), так как HTTP-роутинг и передача параметров компилируются напрямую в оптимальный машинный код без накладных расходов.

## 1. Декораторы маршрутов (Route Decorators)

Синтаксис поддерживает декораторы над функциями для привязки их к HTTP-маршрутам.
Поддерживаемые методы:
- `@get("/path")`
- `@post("/path")`
- `@put("/path")`
- `@delete("/path")`

```rust
@get("/ping")
fn ping() -> string {
    return "pong";
}
```

## 2. Передача параметров (Request Body)

Если обработчик ожидает данные (определен ровно один строковый параметр), компилятор автоматически связывает тело HTTP-запроса с этим аргументом.

```rust
@post("/echo")
fn echo(req_body: string) -> string {
    return req_body;
}
```
*Примечание: в будущих версиях языка (по мере переписывания сервера на самом Rest) планируется поддержка десериализации JSON в структуры.*

## 3. Магия под капотом (Как это работает в LLVM)

Вы можете задаться вопросом: как функция на языке Rest превращается в обработчик HTTP запросов?
Когда компилятор видит декоратор `@get`, он делает три вещи на этапе кодогенерации (`src/codegen/compile.rs`):

1. **Генерация Wrapper-функции (`generate_http_wrapper`)**
   Компилятор создает анонимную C-функцию (wrapper) с сигнатурой:
   `char* __rest_http_wrapper_ping(char* body)`
   
   Внутри этого wrapper'а происходит:
   - Автоматический `__rest_retain(body)` для безопасного удержания переданной строки в памяти.
   - Вызов пользовательской функции `ping()`.
   - `__rest_release(body)` перед возвратом результата.
   - Возврат результата (если он строковый) обратно в HTTP сервер. Если функция `void`, возвращается `null`.

2. **Регистрация в Глобальном Роутере**
   Если в коде найден хотя бы один декоратор, компилятор создает глобальную функцию `__rest_user_main` (которая оборачивает вашу функцию `main`, если она есть) и переименовывает настоящую точку входа в `main`. В настоящем `main` компилятор вызывает системную функцию:
   `rest_register_route("GET", "/ping", &__rest_http_wrapper_ping)`

3. **Запуск сервера**
   В конце сгенерированной `main` функции вставляется вызов `rest_start_server(8080)`, который переводит процесс в бесконечный цикл прослушивания сокетов и обработки запросов.

## 4. Рантайм (Текущий и Будущий)

На данный момент (пока мы не переписали это на Rest) `rest_start_server` реализован в папке `runtime/src/server.rs`.
Это многопоточный асинхронный сервер на основе `Tokio` и `Hyper`. Он использует `Lock-Free` роутер (на базе хэш-таблицы защищенной Rcu / Arc-like подходом), который напрямую вызывает скомпилированные LLVM-указатели на ваши функции. Данные (HTTP body) передаются в Rest без копирования (Zero-Copy string conversion).

**План миграции:** В ближайшем будущем мы перепишем этот сервер на самом Rest, используя FFI и нативные Linux-сокеты (`sys_epoll`). Это удалит зависимость от Tokio и сделает язык полностью независимым.
