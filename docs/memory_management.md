# Управление памятью (ARC - Automatic Reference Counting)

Язык Rest спроектирован так, чтобы разработчик не думал о ручном управлении памятью, при этом сохранялась высокая производительность, сопоставимая с C и Rust.
Мы полностью отказались от старого механизма Borrow Checker-а в пользу неявного и автоматического управления памятью через подсчет ссылок.

## 1. Архитектура ARC

Наш ARC основан на скрытом заголовке (header) счетчика ссылок, который размещается в куче (heap) непосредственно перед полезными данными (payload).

- **Header Size**: 4 байта (`i32`).
- **Счетчик**: Хранит текущее количество владельцев указателя.
- **Payload**: Сами данные (содержимое структуры, элементы массива).
- **Указатели**: Пользовательские функции и переменные работают исключительно с указателем на *payload*. Указатель на начало блока (нужный для системного `free`) вычисляется скрыто смещением на 4 байта назад (`ptr - 4`).

Этот layout относится к **структурам и массивам**. Строки используют другой формат — отдельную 24-байтовую структуру (`ref_count` @0, `len` @8, указатель на данные @16), где сами данные выделяются отдельным `malloc`.

## ⚠️ Известное ограничение: строки текут (не освобождаются)

**На данный момент ARC реализован полностью только для структур и массивов.** Для строк функции `rest_retain_string` и `rest_release_string` являются заглушками (no-op): счётчик ссылок строки не инкрементируется и не декрементируется, а память строки **никогда не освобождается** через `free`.

Последствия:
- Каждая созданная строка (включая результат конкатенации `+`) остаётся в памяти до завершения процесса.
- Для короткоживущих программ это незаметно. Для долгоживущих циклов (например, обработка в бесконечном `while`) это приводит к постоянному росту потребления памяти.

Это осознанный временный компромисс MVP, а не «правильное» поведение. Полная реализация требует аккуратной работы с владением временными строками, чтобы избежать double-free.

## 2. Встроенные Runtime-функции

Компилятор автоматически генерирует в LLVM IR вызовы следующих C-совместимых runtime-функций, которые линкуются прямо в конечный бинарник:

1. `*mut u8 __rest_alloc(i32 size)`:
   - Вызывает системный `malloc(size + 4)`.
   - Инициализирует счетчик ссылок (первые 4 байта) значением `1`.
   - Возвращает указатель на 5-й байт (payload).

2. `*mut u8 __rest_retain(*mut u8 ptr)`:
   - Проверяет указатель на `null`.
   - Вычисляет `real_ptr = ptr - 4`.
   - Атомарно инкрементирует значение счетчика (`*real_ptr += 1`).
   - Возвращает исходный `ptr` для удобства встраивания в выражения (chaining).

3. `void __rest_release(*mut u8 ptr)`:
   - Проверяет указатель на `null`.
   - Вычисляет `real_ptr = ptr - 4`.
   - Атомарно декрементирует счетчик (`*real_ptr -= 1`).
   - Если счетчик падает до `0`, вызывается системный `free(real_ptr)`, тем самым полностью освобождая память. Если освобождаемый объект содержит другие объекты (структура или массив строк), вызывается каскадное освобождение (`rest_free`).
   - **Примечание:** относится к структурам и массивам. Для строк release — заглушка (см. раздел про ограничение выше).

4. `void __rest_free(*mut u8 ptr)`:
   - Принудительно очищает память объекта. Используется сборщиком мусора, когда счетчик `release` достигает нуля.

## 3. Интеграция в Codegen (Трекинг времен жизни)

Автоматическое управление памятью интегрировано напрямую в этап генерации LLVM IR (`src/codegen/compile.rs`). AST освобождено от ручного трекинга времен жизни (lifetimes).

### Присваивания (Assignments)
Когда мы переназначаем переменную или поле структуры (`A = B;`), происходит следующее:
1. Вызов `__rest_release(A)` для старого значения (предотвращение утечек).
2. Вызов `__rest_retain(B)` для нового значения.
3. Сохранение `B` в `A`.

### Выход из Scope (Освобождение)
Компилятор ведет таблицу активных ссылок в текущем скоупе (блок `{}`). 
Для каждой переменной, структуры или массива, которые выходят из области видимости, компилятор добавляет вызовы `__rest_release(ptr)`.

### Глубокая Очистка (Deep Free)
Если структура (object) выходит из области видимости и ее счетчик обнуляется, компилятор рекурсивно освобождает все ее reference-counted поля (подобно `Drop` в Rust, но генерируется динамически на основе типов полей `HirType`). То же самое происходит при освобождении массива строк или массива структур.

## 4. Взаимодействие Аллокатора с FFI

Для системных вызовов и FFI можно использовать обычные указатели. Однако нужно помнить:
- Память, выделенная через системный `malloc` из C-библиотеки, **не должна** передаваться в функции `__rest_release`, так как у нее нет 4-байтового хидера.
- Аналогично, если вы передаете указатель, аллоцированный внутри Rest (через ARC), в системную библиотеку, которая попытается вызвать `free(ptr)`, произойдет сбой (SIGABRT), так как указатель Rest смещен на 4 байта.
- Если требуется полная ручная совместимость, используйте оператор `as` и сырые указатели `*u8`.
